## 2024-05-18 - Client-Side DOM Injection due to Mixed Content Payloads
**Vulnerability:** A critical Cross-Site Scripting (XSS) vulnerability existed because user names, message content, and formatting HTML (`<b>`) were concatenated into single strings on the client *before* being encrypted and broadcast. The receiving clients then directly used `innerHTML` to render these decrypted strings, resulting in code execution if a user included malicious HTML in their message.
**Learning:** Even in applications featuring End-to-End Encryption (E2EE), client-side data handling remains a major attack surface. Encrypting an XSS payload simply ensures it is securely delivered to all clients to be executed.
**Prevention:** Never mix data with presentation logic. Send structured data (like JSON) through the WebSocket layer, and always use safe DOM manipulation methods (`textContent`, `createElement`) on the receiving end to build the UI dynamically, explicitly avoiding `innerHTML`.
